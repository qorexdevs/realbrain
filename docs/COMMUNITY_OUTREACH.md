# Community Outreach Plan

This is an ethical, non-spam growth plan for RealBrain. The goal is to attract builders who care about local-first agent memory, not to manufacture fake popularity.

## Rules

- Do not buy stars, fake engagement, astroturf, or use bots.
- Do not post the same message everywhere.
- Follow each community's self-promotion rules.
- Lead with a useful demo, technical details, or a question.
- Disclose that this is the project author's repo when posting from personal accounts.
- Do not imply RealBrain is conscious, autonomous, or production-safe for secrets.

## Primary audience

1. Agent runtime builders.
2. MCP tool/server builders.
3. OpenClaw users and plugin authors.
4. Obsidian / local-first knowledge-base builders.
5. Python developers interested in lightweight memory infrastructure.
6. Researchers/practitioners thinking about agent memory quality.

## Positioning wedge

Agents usually fail at memory in two ways: they forget everything, or they remember too much badly. RealBrain is the small local layer in between: evidence-linked events, graph memory, global workspace, and suggestion-only consolidation that keeps markdown/Obsidian human-readable.

## Places to share manually

### GitHub

- Add repo topics and public starter issues.
- Enable Discussions.
- Cut a `v0.1.0` release with conservative notes.
- Pin issues for MCP server, OpenClaw plugin, benchmarks, demo assets, and packaging.
- Submit carefully to relevant awesome lists only if they accept PRs and RealBrain fits their criteria.

### Hacker News

Best format: `Show HN: RealBrain – local-first operating memory for AI agents`.

Post once. Be ready to answer technical criticism and avoid hype.

### Reddit

Only post where self-promotion is allowed and the post is useful on its own.

Potential fits:

- r/LocalLLaMA: focus on local-first agent memory and MCP wrappers.
- r/ObsidianMD: focus on markdown/Obsidian as human-readable source of truth.
- r/MachineLearning or r/learnmachinelearning: only if framed as engineering discussion, not marketing.
- agent-builder communities: focus on implementation and safety boundaries.

### LinkedIn

Use a founder/building-in-public angle. Ask for contributors on MCP, OpenClaw, benchmarks, demo assets, and packaging.

### Medium / blog

Publish a technical article, not a launch ad:

- Problem: agents forget or remember slop.
- Model: events → graph → workspace → review queues.
- Safety: hypotheses are not facts; host owns approvals.
- Demo: fake-data walkthrough.
- CTA: contribute examples and wrappers.

### Discord / Slack communities

Share only in channels where project launches are welcome. Keep it short, ask for feedback, and do not cross-post aggressively.

## 7-day launch cadence

Day 1:

- Confirm README and release are ready.
- Post GitHub release.
- Post one technical launch thread on X/LinkedIn.

Day 2:

- Share Show HN if the repo has a clean demo and issues ready.
- Reply quickly to comments.

Day 3:

- Post one technical Reddit thread in the best-fitting community.
- Do not repost identical text elsewhere.

Day 4:

- Publish Medium/blog walkthrough.
- Link it from README if useful.

Day 5:

- Reach out to 5-10 relevant maintainers individually only if there is a real fit, such as MCP/OpenClaw/local-first memory.
- Ask for feedback, not stars.

Day 6:

- Convert feedback into issues and quick fixes.
- Thank contributors publicly where appropriate.

Day 7:

- Publish a small update: what changed, what contributors can pick up next.

## Contribution CTAs

- “Want a good first issue? Start with docs, CLI examples, or demo screenshots.”
- “If you build MCP tools, I’d love feedback on the wrapper shape.”
- “If you use OpenClaw, the plugin issue is the highest-leverage contribution.”
- “If you care about agent memory evals, benchmark fixtures would be extremely helpful.”

## Success metrics

Track useful contribution signals, not just stars:

- unique contributors;
- issue comments with implementation feedback;
- forks with active branches;
- PRs opened;
- demo installs attempted;
- MCP/OpenClaw wrapper interest;
- benchmark ideas.

Stars are a useful discovery signal, but real adoption comes from working examples and credible technical discussion.
